## Why

README.md 已经明确了 Smart-Provider 要解决的问题（客户端高频请求导致上游 API 返回 429、重试风暴、延迟抖动），并提出了“先排队，再放行”的核心思路。然而，项目目前缺少一份系统性的架构设计文档来指导后续实现：各职责模块如何划分、请求如何流入与流出、速率限制如何计算与生效、如何水平扩展以支持更高并发等关键问题尚未定型。因此，需要基于 README.md 的核心目标，输出一份完整且不包含具体代码的项目架构设计方案，为后续编码、测试和部署提供清晰蓝图。

## What Changes

- 新增系统架构总览文档，明确 Smart-Provider 的核心组件及其职责边界。
- 定义请求端到端数据流：从客户端请求接入、队列缓冲、RPM 限速、上游转发，到结果返回。
- 明确关键接口与配置模型（仅描述语义，不写代码）。
- 给出部署与运行视图，说明代理与客户端、上游 API 之间的部署关系。
- 识别后续可扩展点（TPM 限速、优先级队列、熔断、可观测性等），为迭代预留空间。
- 本变更仅产出设计文档，不提交任何可执行代码或测试代码。

## Capabilities

### New Capabilities

- `request-ingress`：定义客户端请求如何进入 Smart-Provider，包括协议适配、请求校验与上下文封装。
- `request-queue`：定义请求队列的存储模型、排队策略、出队条件与生命周期。
- `rate-limiting`：定义 RPM 限速的计算窗口、令牌/漏桶策略选型、并发控制与超限处理。
- `upstream-forwarding`：定义如何按可控速率将请求转发至真实上游 API，并处理响应与错误。
- `response-return`：定义如何将上游响应（含成功、错误、超时）返回给客户端。
- `configuration`：定义运行时配置的结构、加载方式与动态更新能力。
- `observability`：定义需要暴露的指标、日志与追踪维度，以验证限速效果。

### Modified Capabilities

- 无现有 spec 需要修改。

## Impact

- 影响范围：项目根目录下的设计文档与 openspec 变更目录。
- 不引入新的运行时依赖或可执行代码。
- 为后续实现阶段提供唯一参考，减少因理解不一致导致的返工。
